## Fixes aplicados (todos em ZYONCS via apply_migration / cron.unschedule / cron.schedule)

### A) `fn_freshness_watchdog` — qualificar `status` (30 falhas/24h)
Variável local da função (`RETURNS TABLE (... status text ...)`) colidia com coluna `status` no `IF NOT EXISTS (SELECT 1 FROM cs_freshness_alerts WHERE ... AND status = ...)`. Renomeado o filtro interno para usar alias da tabela (`fa.status`).

### B) cron `mv-ocorrencias-refresh-30min` (48 falhas/24h)
Comando antigo: `REFRESH MATERIALIZED VIEW IF EXISTS mv_cli360_data;` — `IF EXISTS` não é válido em REFRESH. Tentei reagendar com `CONCURRENTLY` mas a MV não tem unique index obrigatório. Reagendado como `REFRESH MATERIALIZED VIEW mv_cli360_data;` (sem CONCURRENTLY, sem IF EXISTS). Novo jobid 158.

### C/D) crons órfãos deletados via `cron.unschedule(jobid)` (27 falhas/24h)
- `83` self-heal-on-incidents — `fn_self_heal_incidents()` foi removida
- `32`, `33` menu_amanha_* — `vw_menu_amanha` foi removida
- `76` auto-canonical-pagadores-diario — schema de `cs_pagadores_canonical` mudou (cnpj→cnpj_radical, função não foi atualizada)

### E) `fn_ra_escalar_pendentes` — vault API correta (96 falhas/24h)
Função tentava `vault.create_secret_or_get('EVOLUTION_BASE_URL')` que não existe. API correta no Supabase Vault é via view `vault.decrypted_secrets` — `(SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'EVOLUTION_BASE_URL' LIMIT 1)`. Validado: 0 RAs pendentes vão escalar imediatamente, então a correção não dispara avalanche de WhatsApp para Rubens.

### F) `fn_apply_alert_rules` — pular rules sem `sql_detector` (24 falhas/24h)
3 rules ATIVAS com `sql_detector` NULL: ATRASO_GOLLOG, RUPTURA_GOLLOG, VOLUME_PERDIDO_3D. Geravam `FROM () AS detected` (string vazia interpolada). Adicionado filtro `AND sql_detector IS NOT NULL AND length(trim(sql_detector)) > 0`. Adicionado também `BEGIN ... EXCEPTION WHEN others` por rule, evita que uma rule quebrada derrube o cron inteiro.

### H) `mv-cli360-refresh-1h` timeout — índice direto (24 falhas/24h)
`fn_pagador_canonical` faz `SELECT pagador_documento FROM cs_esl_ocorrencias WHERE pagador_nome = p_nome` (case-sensitive). O índice existente era `idx_esl_ocorrencias_pagador ON tracking.esl_ocorrencias (upper(pagador_nome))` — não cobre o predicado direto, gerava Seq Scan em 41k rows.

Criado `idx_esl_ocorrencias_pagador_nome_doc ON tracking.esl_ocorrencias (pagador_nome) WHERE pagador_documento IS NOT NULL`.

Validação:
- EXPLAIN antes: Seq Scan, 42ms
- EXPLAIN depois: Index Scan, **0,088ms (478× mais rápido)**
- `fn_pagador_canonical('WHIRLPOOL S.A.')`: 10,67ms total
- REFRESH manual de `mv_cli360_data`: completou em &lt;120s (statement_timeout)

`tracking.awbs` JÁ tinha `idx_tracking_awbs_pagador WHERE pagador IS NOT NULL`, não precisava de novo índice.
