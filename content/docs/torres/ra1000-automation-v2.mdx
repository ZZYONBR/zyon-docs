## Funções server-side criadas (migration `cs_ra_automation_v1`)

### `cs_ra_pipeline_briefing() RETURNS text`
Retorna texto formatado do estado das RAs ativas (em_replica/em_treplica counts + lista pendentes + acionamentos pendentes/vencidos). Usado pelo briefing 07h.

### `cs_ra_check_acionamentos_timeout()`
Para cada `ra_acionamentos` com `prazo_resposta_em < now()-5min` e `status='enviado'`:
- Marca como `timeout`
- Se nivel='gestor' → cria novo acionamento `nivel='kevin'` (5547997571032)
- Se nivel='kevin'  → cria novo acionamento `nivel='rubens'` (5547999711598)
- Logs evento `acionamento_timeout_escalation` em historico_log

Roda hourly via pg_cron jobid 147 (`7 * * * *`). Os novos acionamentos ficam com status `pendente_envio` — Mac mini script pega e dispara.

### `cs_ra_registrar_resposta_acionamento(p_destinatario_whatsapp, p_resposta_texto, p_msg_received_at)`
Casa resposta WhatsApp com acionamento pendente daquele número (mais recente, dentro de 24h). Tenta extrair data via regex `DD/MM(/YYYY)?`. Se data firme:
- Update `ra_acionamentos.data_firme_recebida + status='respondido_com_data'`
- Update `cs_ra_reclamacoes.esl_data_entrega = data_firme`
- Log evento `resposta_gestor_capturada`

Se sem data: `status='respondido_sem_data'`.

### `vw_ra_pipeline_status` view
Estado real-time das RAs ativas + último acionamento + minutos desde acionamento.

## Mac mini scripts

### `ra1000-poll-responses.sh` (cada 5 min)
1. Lê `ra_acionamentos` pendentes via REST (`status=enviado, resposta=null, enviado_em<now-12h`).
2. Pra cada um, chama Evolution `POST /chat/findMessages/icaro-claudio` com `where.key.remoteJid=<numero>@s.whatsapp.net&fromMe=false`.
3. Filtra mensagens com `messageTimestamp > enviado_em`.
4. Concatena textos + chama RPC `cs_ra_registrar_resposta_acionamento`.
5. Notifica Rubens via Evolution (`5547999711598`).

Logs: `~/.local/var/log/ra1000-poll.log`

## Anti-padrões aprendidos

- ❌ NÃO tentar `pg_net` para Tailscale 100.x.x.x — só HTTPS público funciona
- ❌ NÃO usar f-strings com escape `\"` em Python 3.9 (SyntaxError)
- ❌ NÃO usar `python3 - <<HEREDOC` quando já há pipe de stdin — separar em arquivo
- ❌ NÃO confiar em `regexp_match` retornar tupla quando há OR de grupos opcionais — usar bloco `BEGIN...EXCEPTION`
- ❌ NÃO esquecer de testar manualmente antes de carregar launchd (vc descobre o syntax error 5min depois)
- ❌ NÃO usar janela de 12h em filtro de pendentes (perde acionamentos antigos sem resposta) — usar 48h. Corrigido em 2026-05-08 19:30 BRT na função `cs_ra_check_acionamentos_timeout` e em `ra1000-poll-responses.sh`.
- ❌ NÃO usar status `pendente_envio` ou `timeout` em `ra_acionamentos` — constraint só aceita `enviado/lido/respondido/expirou`. Use `expirou` para fechar acionamento vencido e `enviado` para o novo (com WhatsApp já disparado).
- ⚠️ Evolution API (`http://100.65.23.99:8080`) NÃO tem docker-compose nem launchd plist no Mac mini. Caiu em algum momento entre 21:12 BRT 07/05 e 19:30 BRT 08/05 e ninguém percebeu. Próximo passo: criar `com.zzyon.evolution.plist` para auto-restart + healthcheck.

- **Bash**: `~/.local/bin/ra1000-in-hours.sh` no Mac mini (exit 0 dentro / exit 1 fora). Usar antes de `curl` Evolution em qualquer script novo.
- **SQL**: `cs_ra_in_business_hours()` no Supabase (boolean). Usar como guard em qualquer função PL/pgSQL que vá criar acionamento.

### Defesa em profundidade
- Poll script Mac mini filtra `nivel=neq.rubens` na query Supabase (acionamento nivel='rubens' significa CEO precisa ser notificado, não que ele vai responder — não polla inbox dele)
- `cs_ra_in_business_hours()` SQL e helper bash conferem isoweekday + hora local BRT
